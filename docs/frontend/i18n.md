# Internationalisation (i18n)

## Vue d'ensemble

Le projet supporte plusieurs langues via deux couches complémentaires :

- **Backend** : `@adonisjs/i18n` détecte la locale, charge les fichiers de traduction, et les expose via `ctx.i18n`
- **Frontend** : le middleware Inertia partage les traductions en props partagées ; un composable `useT()` les consomme dans les composants Vue

Langues actives : **`en`** (défaut) et **`fr`**.

---

## Architecture — flux complet

```
Requête HTTP
    │
    ▼
DetectUserLocaleMiddleware          (router middleware)
    │  détecte la locale (URL /en|/fr ou Accept-Language)
    │  crée ctx.i18n = i18nManager.locale('en'|'fr')
    │  ctx.i18n.localeTranslations = { "app.nav.dashboard": "Dashboard", … }
    │
    ▼
InertiaMiddleware.share()           (server middleware, évalué lazy)
    │  filtre les clés "app.*" de localeTranslations
    │  strip le préfixe "app."
    │  injecte { locale, appT } dans les props partagées Inertia
    │
    ▼
Page Vue (SSR ou client)
    │  usePage().props.locale  → "en"
    │  usePage().props.appT    → { "nav.dashboard": "Dashboard", … }
    │
    ▼
useT() composable
    │  t('nav.dashboard')        → "Dashboard"
    │  t('boats.list.boats')     → "boats"
    │  t('dashboard.urgentMaintenance.dueAt', { date: '2024-06-01' })
    │                            → "Due 2024-06-01"
```

## Fichiers de traduction

```
resources/lang/
  en/
    app.json        ← traductions de l'app (nav, auth, dashboard, boats…)
    marketing.json  ← traductions des pages marketing (pattern différent, voir ci-dessous)
  fr/
    app.json
    marketing.json
```

### Format `app.json`

Objet JSON imbriqué. Le chargeur AdonisJS (`loaders.fs`) aplatit automatiquement les clés :

```json
{
  "nav": {
    "dashboard": "Dashboard"
  },
  "boats": {
    "list": {
      "search": "Search"
    }
  }
}
```

AdonisJS stocke cela en mémoire comme :

```
"app.nav.dashboard"  → "Dashboard"
"app.boats.list.search" → "Search"
```

Le composable `useT()` expose `t('nav.dashboard')` (sans le préfixe `app.`).

### Interpolation de variables

Syntaxe ICU simple : `{nomVariable}` dans les chaînes.

```json
{
  "dashboard": {
    "urgentMaintenance": {
      "dueAt": "Due {date}",
      "dueAtHours": "Due at {hours}h · current {current}h"
    }
  }
}
```

Usage dans un composant :

```vue
{{ t('dashboard.urgentMaintenance.dueAt', { date: ev.dueAt }) }}
```

## Détection de la locale

Référence : `app/middleware/detect_user_locale_middleware.ts`

Ordre de priorité :

1. **Segment d'URL** : `/en/…` ou `/fr/…` → locale extraite du premier segment
2. **Header `Accept-Language`** : meilleure correspondance parmi les locales supportées
3. **Locale par défaut** : `en` (définie dans `config/i18n.ts`)

La locale détectée est accessible côté Vue via `usePage().props.locale` ou via le composable :

```ts
const { t, locale } = useT()
// locale.value → "en" | "fr"
```

---

## Composable `useT()`

Référence : `inertia/composables/useT.ts`

```ts
import { useT } from '~/composables/useT'

const { t, locale } = useT()
```

| Retour   | Type                             | Description                                 |
| -------- | -------------------------------- | ------------------------------------------- |
| `t`      | `(key: string, vars?) => string` | Traduit une clé, retourne la clé si absente |
| `locale` | `ComputedRef<string>`            | Locale courante (`"en"`, `"fr"`…)           |

Si une clé est absente des traductions, `t()` retourne la clé elle-même — jamais une chaîne vide.

---

## Pattern marketing (différent)

Les pages marketing (`/`, `/pricing`) utilisent un pattern **server-side** :
les routes lisent directement `ctx.i18n.t('marketing.home.hero.title')` et passent un objet `t` en prop Inertia.

Ce pattern coexiste avec `appT` mais ne passe pas par `useT()`. Il n'y a pas de conflit.

Référence : `start/routes/marketing.ts`

## Ajouter une nouvelle langue

### 1. Créer les fichiers de traduction

```
resources/lang/de/app.json
resources/lang/de/marketing.json   (si les pages marketing doivent être traduites)
```

Copier `resources/lang/en/app.json` comme base et traduire toutes les valeurs. **Ne pas modifier les clés.**

### 2. Déclarer la locale dans la détection

Référence : `app/middleware/detect_user_locale_middleware.ts`, méthode `getRequestLocale`.

Ajouter `'de'` aux segments URL acceptés :

```ts
if (firstSegment === 'en' || firstSegment === 'fr' || firstSegment === 'de') {
  return firstSegment
}
```

`@adonisjs/i18n` détecte automatiquement les nouvelles locales dès que les fichiers existent dans `resources/lang/de/`.

### 3. Vérifier la locale par défaut

Référence : `config/i18n.ts`

```ts
const i18nConfig = defineConfig({
  defaultLocale: 'en',   // inchangé
  …
})
```

### 4. Aucun changement côté Vue

Le middleware Inertia et `useT()` sont agnostiques au nombre de langues. Une fois les fichiers créés et la détection mise à jour, les composants Vue reçoivent automatiquement les bonnes traductions.

---

## Ajouter des clés de traduction

1. Ajouter la clé dans `resources/lang/en/app.json`
2. Ajouter la traduction dans **tous** les autres fichiers `app.json` existants (`fr`, etc.)
3. Utiliser la clé dans le composant Vue :

```ts
const { t } = useT()
// template : {{ t('ma.nouvelle.cle') }}
```

Les clés manquantes dans une locale font que `t()` retourne la clé brute — aucune erreur runtime, mais ça doit être détecté en review.

## Structure des clés `app.json`

| Namespace                    | Contenu                               |
| ---------------------------- | ------------------------------------- |
| `common.*`                   | Oui/Non, Annuler, Supprimer, Fermer…  |
| `nav.*`                      | Navigation latérale et header         |
| `auth.login.*`               | Page de connexion                     |
| `auth.signup.*`              | Page d'inscription                    |
| `dashboard.*`                | Dashboard, stats, maintenance urgente |
| `boats.index.*`              | Page liste des bateaux                |
| `boats.show.*`               | Page détail d'un bateau               |
| `boats.list.*`               | Toolbar, tableau, cartes, pagination  |
| `boats.maintenance.events.*` | Panel historique de maintenance       |
| `boats.maintenance.tasks.*`  | Panel tâches planifiées               |
