## Bugs HIGH

### A-01

**[Auth / Invitations] N'importe quel user peut accepter l'invitation d'un autre**

- **Fichier :** `app/services/organization_invitation_service.ts` lignes 160–186
- **Comportement actuel :** `accept()` vérifie uniquement qu'il n'y a pas déjà une membership pour le `userId`, mais ne compare jamais `user.email` avec `invitation.email`. Tout utilisateur authentifié qui possède ou dépasse l'URL du lien d'invitation peut l'accepter à la place du destinataire.
- **Comportement attendu :** Rejeter avec `InvitationEmailMismatchError` si `user.email.toLowerCase() !== invitation.email.toLowerCase()`.
- **Impact :** Escalade de privilège horizontale — Bob accepte l'invitation d'Alice et obtient son rôle dans l'organisation d'Alice.
- **Piste de fix :** Charger le `User` via `userId` dans `accept()`, comparer les emails avant de créer la membership.

---

### A-02

**[Auth / Invitations] `accept()` ne re-scope pas le user vers la nouvelle org**

- **Fichier :** `app/services/organization_invitation_service.ts` lignes 174–178
- **Comportement actuel :** La `OrganizationMembership` est créée pour la nouvelle org, mais `user.organizationId` en base reste inchangé. Tous les controllers scopent les données via `user.organizationId` (ex: `organization_members_controller.ts` lignes 29, 46, 83, 110). Après acceptation, l'utilisateur voit encore les données de son org d'origine.
- **Comportement attendu :** Mettre à jour `user.organizationId = invitation.organizationId` + `user.save()` dans la même transaction que la création de membership.
- **Impact :** La feature d'invitation est fonctionnellement inopérante pour le re-scoping de l'utilisateur.
- **Piste de fix :** Dans `accept()`, charger le User, mettre à jour `organizationId`, sauvegarder en transaction.

### A-03

**[Auth / Org] Signup ne crée pas de membership → owner bloqué pour gérer ses membres**

- **Fichier :** `app/services/user_service.ts` (ou `organization_service.ts`) — `signupWithOrganization()`
- **Comportement actuel :** `createForSignup()` crée l'org et pose `user.organizationId`, mais ne crée aucune row `organization_memberships`. Or `User.isAdminOf()` (`app/models/user.ts` lignes 48–51) ne consulte que la table memberships. Le hook `before()` d'`OrganizationPolicy` appelle `isAdminOf()` → retourne `false` pour le créateur. Résultat : le propriétaire fraîchement inscrit reçoit 403 sur toute action `manageMembers` (inviter, annuler, changer le rôle).
- **Comportement attendu :** `signupWithOrganization()` doit créer une `OrganizationMembership` avec `role: 'admin'` dans la même transaction.
- **Impact :** Bloquant — aucune org créée via signup ne peut inviter de membres. La feature invitation est inutilisable depuis l'UI.
- **Piste de fix :** Après `User.create()`, ajouter `OrganizationMembership.create({ userId, organizationId, role: 'admin' }, { client: trx })`.

### B-01

**[Media] IDOR : `deleteById` sans vérification d'entité ni d'org**

- **Fichier :** `app/services/media_service.ts` ligne 108–130
- **Comportement actuel :** `deleteById(mediaId, org)` fait un `Media.find(mediaId)` (lookup bare par PK) sans vérifier que le média appartient à l'entité ou à l'org du contexte. Le paramètre `org` est uniquement utilisé pour décrémenter le quota. Un user de l'org A qui a le droit d'éditer un bateau peut supprimer n'importe quel media ID en le passant dans la route `/boats/:id/media/:mediaId`.
- **Comportement attendu :** `deleteById` doit vérifier que `media.entityType` et `media.entityId` correspondent à l'entité attendue, ou que l'org du media correspond à l'org du contexte.
- **Impact :** IDOR — suppression cross-org de fichiers. Corruption du quota de l'org appelante.
- **Piste de fix :** Ajouter un paramètre `entityType / entityId` à `deleteById` et filtrer comme dans `getForEntity()`, ou créer une méthode dédiée `deleteForEntity(mediaId, entityType, entityId, org)`.

### B-02

**[Media / PDF] PDFs non supprimés sur Cloudinary (resource type inversé)**

- **Fichier :** `app/services/media_service.ts` ligne 19
- **Comportement actuel :** `resourceTypeFromKind()` retourne `'image'` quand `format === 'pdf'`. Mais le code de téléchargement (`boat_media_controller.ts` lignes 219, 255) utilise correctement `format === 'pdf' ? 'raw' : 'image'`. La condition est inversée : les PDFs sont uploadés comme `'raw'` mais supprimés comme `'image'` → Cloudinary renvoie une erreur silencieuse ou no-op.
- **Comportement attendu :** `if (format === 'pdf') return 'raw'` (inverser la condition).
- **Impact :** Tout document PDF supprimé via l'UI reste physiquement sur Cloudinary (fuite de stockage permanente). Le quota DB est décrémenté correctement, créant une divergence croissante quota/stockage réel.
- **Piste de fix :** Inverser la condition ligne 19 : `if (kind === 'photo') return 'image'` / `if (format === 'pdf') return 'raw'` / `return 'raw'`.

### B-03

**[Équipements] IDOR sur media de pièces moteur**

- **Fichier :** `app/controllers/boat_engine_parts_controller.ts` lignes 202–267 (`destroyMedia`, `downloadMedia`)
- **Comportement actuel :** Les deux actions ne valident pas la chaîne `partId → engineId → boatId → org`. Seul le `boatId` est vérifié via `loadBoat`. Un user de l'org A peut forger l'URL `/boats/:MON_BATEAU/engines/:N_IMPORTE_QUEL_MOTEUR/parts/:PART_ORG_B/media/:MEDIA_ID` pour lire ou supprimer des documents d'une autre org.
- **Comportement attendu :** Vérifier `engine = boat.engines.find(e.id === engineId)` puis `part = findForEngine(engineId, partId)` avant toute opération — comme c'est fait dans `storeDocument` (lignes 176–180).
- **Impact :** IDOR cross-org — lecture et suppression de fichiers d'autres organisations, corruption du quota.
- **Piste de fix :** Ajouter les mêmes guards que dans `storeDocument` au début de `destroyMedia` et `downloadMedia`.

### C-01

**[Réservations] PATCH partiel efface les champs optionnels**

- **Fichier :** `app/controllers/boat_reservations_controller.ts` lignes 114–123
- **Comportement actuel :** Le controller convertit les champs absents (`undefined`) en `null` via `?? null` avant de les passer au service (`clientEmail: payload.clientEmail ?? null`). Le service protège ces champs via `if (payload.clientEmail !== undefined)` — mais `null !== undefined` est `true`. Le service croit que le champ est intentionnellement mis à null et l'écrase.
- **Comportement attendu :** Passer les champs directement sans `?? null` pour préserver `undefined` et laisser les guards du service fonctionner.
- **Impact :** Data corruption — tout PATCH qui ne renvoie pas tous les champs (toggle de statut, changement de dates) détruit clientEmail, clientPhone, notes et totalPrice.
- **Piste de fix :** Retirer `?? null` sur les 4 champs optionnels dans `update()`.

### D-01

**[Navigation / Équipage] IDOR crew sync : crew IDs non scopés à l'org**

- **Fichier :** `app/services/crew_service.ts` lignes 110–119
- **Comportement actuel :** `syncCrewForNavigationLog` construit `pivotData` directement depuis `payload.crew` sans vérifier que les IDs appartiennent à la même org que le log. La pivot `navigation_log_crew` n'a pas de colonne `organization_id`. Un user de l'org A peut injecter des IDs de membres de l'org B dans son log.

- **Impact :** Pollution cross-org — membres d'une autre organisation apparaissent dans les logs d'une org tierce.
- **Piste de fix :** Requête préalable de vérification + rejeter/ignorer les IDs inconnus.

### D-02

**[Équipage] Certification `expiresAt` cassée en PostgreSQL**

- **Fichier :** `app/controllers/crew_certifications_controller.ts` ligne 29
- **Comportement actuel :** VineJS `vine.date()` retourne un objet `Date` JS. Le controller fait `String(payload.expiresAt)` = `Date.prototype.toString()` → `"Mon Jan 15 2024 00:00:00 GMT+0000 ..."`. Cette chaîne non-ISO est passée au service qui l'injecte dans un champ `@column.date()`. PostgreSQL lève `invalid input syntax for type date` → exception 500 systématique.
- **Comportement attendu :** `DateTime.fromJSDate(payload.expiresAt).toISODate()` → `"2024-01-15"`.
- **Impact :** La création de certification est **cassée en production PostgreSQL** — 100% des créations échouent avec 500.
- **Piste de fix :** Remplacer `String(payload.expiresAt)` par `DateTime.fromJSDate(payload.expiresAt).toISODate()` ; retirer le cast `as unknown as any` dans le service.

### F-01

**[Budget / Escales] `endedAt` peut être antérieur à `startedAt`**

- **Fichier :** `app/validators/boat_port_stay_validator.ts` ligne 7
- **Comportement actuel :** VineJS valide `startedAt` et `endedAt` indépendamment. Un payload `{ startedAt: "2025-12-01", endedAt: "2025-01-01" }` passe le validator et est persisté avec `ended_at < started_at`.
- **Comportement attendu :** Validation cross-field `.afterOrSameAsField('startedAt')` sur `endedAt`.
- **Impact :** Données incohérentes en base, calculs de durée négatifs.
- **Piste de fix :** Ajouter une règle `vine.date().afterField('startedAt').optional()` ou un `refine` custom sur l'objet.

### F-02

**[Documents] Document marqué "expiré" toute la journée J**

- **Fichier :** `app/services/boat_document_service.ts` lignes 18–20 (`computeStatus`)
- **Comportement actuel :** `expiresAt` est une colonne `date` (minuit). `DateTime.now()` inclut l'heure courante. Si `expiresAt = 2025-06-29 00:00:00` et qu'il est 14h00 le 29 juin, alors `diff = -0.58 jours < 0` → statut `"expired"` pendant toute la journée du 29.
- **Comportement attendu :** Comparer en jours entiers : `Math.floor(expiresAt.startOf('day').diff(DateTime.now().startOf('day'), 'days').days)`.
- **Impact :** Faux positifs — tous les documents affichent "expiré" le jour même de leur expiration.
- **Piste de fix :** Normaliser les deux dates à `startOf('day')` avant le diff.

### F-03

### G-02

- `app/services/subscription_service.ts` lignes 85–104. `getPeriodBounds()` recalcule manuellement depuis `billing_cycle_anchor` au lieu d'utiliser `stripeSub.current_period_start` / `current_period_end` (faisant autorité).

### A-08

**[Billing] Sync abonnement non atomique**

- `app/services/subscription_service.ts` lignes 35–63. `upsertSubscription()` puis `updateOrgPlan()` en séquence sans transaction. En cas d'échec partiel : subscription `active` mais org encore sur `starter`.

### A-09

**[Invitations] "Déjà membre" ne couvre pas l'owner**

- `app/services/organization_invitation_service.ts` lignes 55–64. La check "déjà membre" ne consulte que `organization_memberships`. L'owner sans membership row peut recevoir une invitation à sa propre org. Se corrige après fix de A-03.

### B-04

**[Bateaux] `registrationNumber` unique → 500 non géré**

- `app/controllers/boats_controller.ts` lignes 104–105, 283–284. Contrainte unique DB `(organization_id, registration_number)` non interceptée → 500 brut en cas de doublon.

### B-05

**[Équipements] Heures moteur peuvent régresser**

- `app/services/boat_engine_service.ts` ligne 52. `engine.hours = payload.hours ?? null` sans vérifier `payload.hours >= engine.hours`. Manipulation possible du compteur.

### B-06

**[Bateaux] `assign()` sans `bouncer.authorize()`**

- `app/controllers/boats_controller.ts` lignes 327–346. Tout membre authentifié peut réassigner n'importe quel bateau de l'org. Toutes les autres actions destructives (update, destroy) appellent bien la policy.

### C-02

**[Réservations] `option` bloque les `confirmed` (pas de priorité)**

- `app/services/boat_reservation_service.ts` lignes 151–154. `whereNot('status', 'cancelled')` : les `option` ont le même pouvoir bloquant que les `confirmed`. Workflow bloqué si deux options couvrent la même période.

### C-03

**[Réservations] Pas de machine à états sur les transitions de statut**

- `app/services/boat_reservation_service.ts` ligne 122. Transitions invalides non bloquées : `confirmed → option`, `cancelled → confirmed`. N'importe quel membre peut dégrader un booking ferme.

### D-03

**[Navigation] Payload crew vide supprime silencieusement tout l'équipage**

- `app/services/crew_service.ts` ligne 118. Si `payload.crew = []`, `sync({})` détache tous les membres existants. Pas de garde `.minLength(1)` dans le validator.

### D-04

**[Navigation / Crew] `vine.compile()` dans 7 validators**

- `app/validators/crew.ts` (4 validators) et `app/validators/navigation_log.ts` (3 validators). Remplacer par `vine.create()`.

### D-05

**[Navigation] Update log partiel efface les champs optionnels**

- `app/controllers/navigation_logs_controller.ts` lignes 80–85. Même pattern que C-01 : `payload.windForceBeaufort ?? null` convertit `undefined → null`, déjouant les guards `!== undefined` du service. Champs effacés sur chaque update partiel.

### D-06

**[Navigation] `closeTrip` met à jour le mauvais moteur**

- `app/services/navigation_log_service.ts` lignes 140–145. `orderBy('id', 'asc').first()` — sélectionne arbitrairement le premier moteur actif. Aucun champ `boatEngineId` sur `NavigationLog` pour indiquer le moteur utilisé.

### D-07

**[Navigation] Aucun guard contre 2 logs `in_progress` pour le même bateau**

- `app/services/navigation_log_service.ts` lignes 48–73. Pas de check préalable ni d'index partiel unique sur `(boat_id, status) WHERE status = 'in_progress'`.

### D-08

**[Dates] `toDateTime()` sans normalisation UTC**

- `shared/helpers/date.ts` ligne 6. `DateTime.fromJSDate(value)` sans `{ zone: 'UTC' }` crée un DateTime dans la zone locale du serveur. Datetimes stockés dépendent du TZ du process (prod Docker UTC vs. dev Mac local).

### E-03

**[CSV Import] `vine.compile()` dans csv_import validators**

- `app/validators/csv_import.ts` lignes 3 et 14. Remplacer par `vine.create()`.

### E-04

**[Maintenance] `completeSheet()` ne vérifie pas que tous les items sont cochés**

- `app/services/boat_maintenance_sheet_service.ts` lignes 90–104. Une sheet avec zéro items cochés peut être marquée "complétée".

### E-05

**[Maintenance] Items d'une sheet complétée modifiables a posteriori**

- `app/services/boat_maintenance_sheet_service.ts` lignes 127–157. `updateItem()` ne vérifie pas `sheet.status === 'completed'`. Les fiches "terminées" sont mutables.

### E-06

**[Maintenance] Décrémentation stock en read-modify-write (race condition)**

- `app/services/boat_maintenance_service.ts` lignes 199–216. Pattern lecture + `Math.max(0, stock - used)` + save en READ COMMITTED : deux transactions concurrentes peuvent toutes deux lire le même stock avant écriture.

### F-04

**[Budget / Escales] `cost = 0` rejeté pour les escales**

- `app/validators/boat_port_stay_validator.ts` ligne 8. `vine.number().positive()` exclut 0. Impossible d'enregistrer une escale gratuite (mouillage, port partenaire). `boat_document.ts` utilise déjà `.min(0)` — incohérence.

### F-05

**[Budget] Documents avec `issuedAt = null` exclus du budget**

- `app/services/budget_service.ts` lignes 55–64. `fetchDocumentsByMonth` filtre sur `EXTRACT(YEAR FROM issued_at) = ?`. Si `issued_at` est `NULL`, PostgreSQL retourne `NULL` → la ligne est exclue même si `cost > 0`.

### F-06

**[Budget] Montant négatif rejeté (avoir/remboursement impossible)**

- `app/validators/budget_entry_validator.ts` ligne 8. `vine.number().positive()` — aucun avoir ni remboursement ne peut être saisi. `SUM(amount)` dans `fetchEntriesByMonth` ne peut jamais diminuer via un crédit.

### G-03

**[Ports] Flash messages hardcodés dans 3 controllers**

- `app/controllers/ports_controller.ts` ligne 103, `pontoons_controller.ts` ligne 73, `mouillages_controller.ts` ligne 75. `session.flash('error', 'port_has_boats')` etc. — clés brutes non traduites.

### G-04

**[Ports] Mauvaise action Bouncer (`create` pour `destroy`/`edit`/`updatePosition`)**

- `app/controllers/pontoons_controller.ts` et `mouillages_controller.ts`. `authorize('create')` utilisé pour toutes les mutations. Si la policy évolue pour autoriser un rôle sur `edit`, le `destroy` hériterait silencieusement de cette permission.

### G-05

**[Ports / Spots] Race condition check-then-delete sur suppression pontoon/mouillage**

- `app/services/pontoon_service.ts` lignes 57–66. Le count de bateaux puis la suppression ne sont pas dans une transaction → un bateau assigné entre les deux opérations perd son spot silencieusement.

### H-02

**[IA] TOCTOU sur le quota de tokens IA**

- `app/services/ai_analysis_service.ts` lignes 120–170. `getUsage()` → `assertCanUseTokens()` → appel API Mistral → `recordUsage()`. Deux requêtes simultanées peuvent toutes deux passer la vérification avant que l'une enregistre sa consommation.

### H-03

**[IA] `AuthorizationException` avalée dans `boatSuggestions`**

- `app/controllers/ai_controller.ts` lignes 104–169. Le `catch` générique avale les `AuthorizationException` Bouncer et produit un flash "erreur d'analyse" au lieu d'un 403. Une restriction de policy serait masquée.
